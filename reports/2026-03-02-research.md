For SailorSkills marketplace, adding WebMCP to marketplace.sailorskills.com means AI agents can search for providers, book services, and query availability — instantly making the marketplace agent-accessible.
2. **"When does MCP make sense vs CLI?" (366 pts, 230 comments) settles the debate: CLI wins for developer tools, MCP wins for non-CLI-native services.** The post argues MCP provides no real-world benefit over CLI for most developer workflows — and the HN consensus largely agrees for coding agents. But the nuance matters: MCP's value is in wrapping non-CLI services (Jira, wikis, databases, internal tools) that agents can't reach via shell commands. The practical takeaway: don't replace working CLIs with MCP servers, but do wrap API-only services.

## 🛠️ Tools & Methods

### MCP vs CLI — Decision Framework for Agent Tooling
- **Source:** https://news.ycombinator.com/item?id=47208398 (366 pts, 230 comments)
- **What it does:** The post (and extensive HN debate) establishes a clear framework: use CLI when the tool already exists and the agent can figure out flags from --help output; use MCP when wrapping non-CLI services (APIs, databases, internal tools) or when you need to control context window consumption, paginate results, or restrict dangerous operations. Key quote: "MCP blew up in 2024 before terminal agents blew up in early 2025. The story isn't 'MCP was fake' — it's how quickly the meta evolves."
- **How we'd use it:** For Brian's OpenClaw setup, this confirms the current approach: skills (CLI-native) for most things, MCP servers only for services that don't have good CLIs (like Jira, Confluence, or proprietary APIs). The Jira MCP discussion is especially useful — multiple commenters report better results having the LLM call the Jira API directly and write scripts than using the official MCP.

### Timber — "Ollama for Classical ML Models"
- **Source:** https://github.com/kossisoroyce/timber (110 pts, Show HN)
- **What it does:** AOT compiler that turns XGBoost, LightGBM, scikit-learn, CatBoost, and ONNX tree-based models into native C99 inference code. 336x faster than Python inference. One command to load, one to serve. Ollama-style workflow for classical ML.
- **How we'd use it:** If SailorSkills builds predictive models for service scheduling (e.g., predicting hull fouling rates based on water temperature, season, anti-fouling paint type), Timber could deploy those models at microsecond latency with zero Python overhead — running directly on the Pro app's backend or even edge-deployed.

### Simon Willison: Claude Memory Import is Just a Prompt
- **Source:** https://simonwillison.net/2026/Mar/1/claude-import-memory/
- **What it does:** Simon discovered that Anthropic's claude.com/import-memory "feature" is literally just a prompt you paste into another AI. It asks the AI to list every stored memory, preference, and instruction. The output goes into Claude's memory settings. No API, no sync, no magic — just prompt engineering as a product.
- **How we'd use it:** Template for any "import from competitor" flow. If Brian builds AI tools for SailorSkills clients, a "switch from [competitor] to SailorSkills" workflow that imports their data via structured prompts is a low-effort, high-conversion growth tactic. Also: the fact that AI memories are this portable means users who invest in OpenClaw's memory system aren't locked in — and that's a selling point.
